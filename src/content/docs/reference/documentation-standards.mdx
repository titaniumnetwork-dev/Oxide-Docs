# Frontmatter Standards

All MDX files use YAML frontmatter for metadata:

## Basic Page Frontmatter

```yaml
---
title: Page Title
description: Brief description of the page content
keywords: ["keyword1", "keyword2", "keyword3"]
---
```

## Proxy/Technology/Transport Pages

```yaml
---
title: Project Name
description: Brief description
keywords: ["keyword1", "keyword2"]
themeColor: "#ffa660"  # Optional: brand color
github: "https://github.com/org/repo"
official-link: "https://example.com"
order: 1  # Optional: sidebar ordering
---
```

## Organization Pages

```yaml
---
title: Organization Name
sidebar:
  order: 1  # Position in sidebar
category: [organizations]
github: "https://github.com/org"
official-link: "https://example.com"
---
```

# Writing Style

## Tone

- **Professional but approachable**
- **Clear and concise**
- **Technically accurate**
- **Community-focused**

## Voice

- Use active voice: "Scramjet provides" not "is provided by"
- Direct address: "You can install" not "One can install"
- Present tense for current features
- Be specific, not vague

## Technical Terms

- **First mention**: Full name with abbreviation in parentheses
- **Subsequent mentions**: Use abbreviation or short form
- **Example**: "Service Worker (SW)" â†’ later use "SW"

### Example from Scramjet docs:

> "In the proxy community, you may see it abbreviated as 'SJ'."

# Structure Patterns

## Page Organization

### 1. Opening Section

- Brief description (1-2 sentences)
- Key features or value proposition
- Context about the project's role

### 2. Feature List

Use bullet points with **bold headings**:

```markdown
- **Feature Name**: Brief description
- **Another Feature**: Brief description
```

### 3. Installation/Setup

````markdown
# Installation

```bash
npm install package-name
````

## Dependencies

List required packages...

````

### 4. Usage Examples
- Start simple, progress to complex
- Include both code and explanations
- Show real-world use cases

### 5. Advanced Topics
- Separate advanced features into subsections
- Use clear hierarchical headings

### 6. Related Links
End with links to:
- GitHub repository
- Official documentation
- Related projects
- Community resources

## Heading Hierarchy

```markdown
# H1: Main Title (Page Title)
## H2: Major Sections
### H3: Subsections
#### H4: Detailed Breakdowns
````

**Rules:**

- Only one H1 per page (the title)
- Don't skip heading levels
- Keep headings concise (3-7 words)
- Use sentence case, not title case

# Admonitions (Callout Boxes)

## Note

For general information or tips:

```markdown
:::note
In the proxy community, you may see it abbreviated as "SJ".
:::
```

## Caution/Warning

For important warnings:

```markdown
:::caution
Fixed types haven't landed in the NPM package yet.
:::
```

## Note with Title

```markdown
:::note[Custom Title]
Content goes here.
:::
```

## When to Use

- **Note**: Non-critical information, helpful tips
- **Caution**: Important limitations or gotchas
- **Warning**: Critical issues that could cause problems
- **Tip**: Best practices or recommendations

# Formatting Best Practices

## Emphasis

### Bold

Use for:

- **Feature names**
- **Important terms** on first use
- **Key concepts**
- **Section emphasis**

### Italic

Use sparingly for:

- *Emphasis within a sentence*
- *Book/document titles*

### Bold + Italic

Use very rarely for ***critical warnings***

## Lists vs. Paragraphs

- Use lists for: Multiple related items, steps, options
- Use paragraphs for: Explanations, descriptions, narrative

## Code vs. Text

- Use code formatting for anything a user would type or see in code
- Use regular text for concepts and explanations

# Consistency Checklist

When writing documentation:

- [ ] Frontmatter includes all relevant fields
- [ ] Page has clear introduction
- [ ] Code examples are formatted with language tags
- [ ] Internal links use relative paths with `.mdx`
- [ ] External links use descriptive text
- [ ] Headings follow hierarchy (no skipped levels)
- [ ] Lists use consistent formatting
- [ ] Technical terms defined on first use
- [ ] Bold used for emphasis, not italic overuse
- [ ] Admonitions used appropriately
- [ ] File names and code in backticks
- [ ] Examples progress from simple to complex
- [ ] Related links at the end
- [ ] Spelling and grammar checked
- [ ] Tone is consistent with site
